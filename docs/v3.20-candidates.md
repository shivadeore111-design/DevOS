## v3.19 C3 findings — added to v3.20 backlog

### 10. Provider rate-limit guard for development (~150 LOC)
Discovered during C3 testing: all 5 groq keys share one org-level 100k TPD pool.
Router uses 15s RPM window — groq gets retried immediately after TPD exhaustion,
starving Gemini. Needs:
- Per-provider TPD budget displayed in CLI/health endpoint
- `AIDEN_DEV_MODE=1` flag to prefer Ollama over cloud
- Router respects per-provider `rateLimitWindow` for cooldown (not always 15s RPM)

### 11. Fast-path pattern audit (~50 LOC + test file)
`matchFastPath` was classifying short action intents (≤19 chars, no exec keywords)
as conversational fast-path, bypassing the planner. Fixed in C3 by adding
`isActionIntent` check first. But the full FAST_PATH_PATTERNS / PLANNER_REQUIRED_PATTERNS
sets need a systematic audit with a 30+ message test suite.

## v3.19 C4 findings — added to v3.20 backlog

### 12. Router TPD-aware routing (~80 LOC)
Discovered during C4: router retries a provider immediately after TPD exhaustion
(daily token budget blown) because `markRateLimited()` only sets a 15s RPM window,
not a 24h TPD window. Needs:
- Separate `markTpdExhausted(provider)` with 24h cooldown stored in config
- Router skips TPD-exhausted providers for the rest of the calendar day
- `/api/health` and CLI `--status` expose per-provider TPD remaining / reset time

### 13. Provider budget visibility (~150 LOC)
Users have no visibility into which providers are healthy, rate-limited, or TPD-exhausted.
Needs:
- `GET /api/providers/status` — returns per-provider health, usage count, RPM/TPD state
- Dashboard "Providers" tab consuming that endpoint
- CLI `aiden status` command printing a provider table
