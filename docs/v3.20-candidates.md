# v3.20 Candidates

Features deferred from v3.19 — assessed, scoped, and held back for focused reasons.

---

### browser_tabs tool (deferred to v3.21+)

**What it would do:** Return the titles and URLs of open tabs in Chrome/Edge/Firefox.

**Why deferred:**
1. CDP approach requires Chrome/Edge launched with `--remote-debugging-port=9222` — a flag most users never set. Zero fallback when the port is closed.
2. Needs a per-session consent UX: exposing tab URLs is sensitive (banking, personal accounts). A raw dump on first ask is a privacy violation.
3. Out of scope for v3.19 "ALIVE" theme — `now_playing` already demonstrates the lazy-state pattern. Adding a half-baked `browser_tabs` dilutes the ship.

**Estimated scope when ready:** ~400–600 LOC including permission system and CDP fallback (history file parsing via SQLite).

**Prerequisite:** Decide on opt-in model (env var flag? runtime consent dialog?) before implementation.

### CLI cursor rendering bug

**Symptom:** Cursor drifts behind text while typing in some terminal contexts. Visible during normal typing — not specific to dropdown trigger.

**Root cause hypothesis:** Interaction between the live dropdown ANSI escape sequences introduced in commit `8c99643` and ConHost/Windows Terminal cursor position restore. The dropdown render loop issues `\r` + partial-line overwrites; when the terminal's cursor-save/restore state is already dirty from a prior render, the next keypress lands the cursor at the wrong column.

**Scope:** ~50 LOC in the dropdown render section of `cli/aiden.ts`. Likely a missing `\x1b[u` / `\x1b[s` flush or an off-by-one on line-count tracking used for the ANSI cursor-up sequence.

**Why deferred:** Does not affect correctness or data — cosmetic only. Reproduces in ConHost and some Windows Terminal configs; not universal. Fixing blind risks destabilizing the dropdown on terminals where it currently works.

**Prerequisite:** Reproduce deterministically in a controlled terminal snapshot, then fix with a targeted ANSI sequence audit.

### system_state tool (stretch goal from phase4-state-audit.md)

**What it would do:** On-demand snapshot of open windows + RAM + disk — replacing the volatile startup injection that was removed in v3.19 C3.

**Why deferred:** `shell_exec` already handles ad-hoc queries. A dedicated `system_state` tool is a clean-up, not a blocker. The startup injection removal (C3) is the critical fix; the new tool is polish.

**When to build:** Bundle with browser_tabs or whenever a common pattern of multi-field system queries emerges.
