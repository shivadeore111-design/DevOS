## [mid-level] Per-session AbortController — concurrent session interrupt safety

**Status check:** `core/agentLoop.ts:66` — `let currentAbortController: AbortController | null = null` — single module-level variable. `interruptCurrentCall()` at `:69` aborts and nulls it. A second concurrent session (API + TUI, Discord + browser chat) overwrites the variable — only the last-started request holds a live controller. `soulHashBySession` at `:63` is already `Map<string, string>` proving the per-session pattern is known; the abort state was not given the same treatment.

**Description:** Single-session interrupt works correctly — a new message cancels the in-flight inference. But when two sessions run concurrently (e.g., a `/api/chat` call from Open WebUI while the TUI is mid-reply), the second session's controller overwrites the first. The first session can no longer be interrupted and `executionInterrupted` is shared state across both. The fix is the same `Map<sessionId, …>` pattern already used for `soulHashBySession`.

**Acceptance criteria:**
- `currentAbortController` replaced with `Map<string, AbortController>` keyed by `sessionId`
- `interruptCurrentCall(sessionId)` aborts only the targeted session's controller
- `executionInterrupted` becomes per-session state (map or passed-through context)
- Existing single-session callers (TUI, which has no explicit sessionId) continue to work via a default session key
- Concurrent API + TUI sessions each independently interruptible with no cross-session interference

**Pointers:**
- `core/agentLoop.ts:63–75` — `soulHashBySession` Map directly above is the model to follow
- `api/server.ts` — passes `sessionId` into `agentLoop`; thread it through to the new interrupt map
- `interruptCurrentCall()` is exported and called from the `/api/stop` endpoint — update its signature

**Estimated effort:** M
